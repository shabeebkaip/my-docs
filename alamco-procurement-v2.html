Step-by-Step Flow

What Happens — Inside Every Step

A granular breakdown of exactly what the system does at each node of the architecture — from the moment a file is uploaded to the moment a status alert fires.

📁
Node 1 · Entry Point
BOQ File Upload
The process begins the moment Alamco's procurement team uploads a Bill of Quantities file. The system accepts any structured format — no special template required.
📤
File Ingestion & Validation
The upload interface accepts Excel (.xlsx, .xls), PDF, CSV, or Word documents. The system immediately validates the file — checking for readability, format integrity, and minimum required structure (at least one data sheet or page with line items). Corrupted or empty files are rejected with a clear error before any processing begins.
🔍
Format Detection
The system auto-detects the file type and routes it to the correct parser: structured Excel parser for spreadsheets (handles merged cells, multi-sheet workbooks, hidden rows), PDF/OCR extractor for scanned documents, or the AI extraction pipeline for non-standard or legacy layouts that don't follow predictable column structures.
🏷️
Project Metadata Capture
Alongside the file, the user optionally provides project metadata: project name, reference number, client, site location, and target delivery date. This context is attached to every procurement record generated downstream, creating a full audit trail from file to purchase order.
🔐
Secure Transfer to AI Agent
Once validated, the file and metadata are packaged and dispatched over an encrypted connection to the AI Procurement Agent. A job is created in the processing queue and the user sees a real-time status indicator. No data is stored in third-party systems — processing happens within Codeox's private infrastructure.
Output of this step: A validated, format-detected file securely queued for AI processing — with project context attached and a unique job ID for tracking.
Supported: .xlsx .xls .pdf .csv .docxAuto format detectionEncrypted transferAudit job ID created
🧠
Node 2 · AI Core · Primary data flow
AI Procurement Agent — Reading & Parsing
This is the intelligence core of the system. The AI agent reads the BOQ file as a senior procurement engineer would — understanding context, inferring division scope, and extracting every actionable procurement item.
🗂️
Division & Section Recognition
The AI identifies the BOQ's division structure — Civil, MEP, Finishing, Fit-out, Furniture, FF&E, and others. It maps the hierarchy: Division → Section → Line Item. Even irregular or non-standard BOQ layouts (no consistent column headers, mixed languages, numerical codes without descriptions) are handled through the AI's contextual reasoning rather than rigid pattern matching.
📋
Line Item Extraction
For each line item the agent extracts: item description (raw text), quantity (numeric), unit of measure (m², m, nr, kg, set, lot…), division/section assignment, and any specification codes or notes referenced. Items that span multiple rows (continuation lines) are merged. Subtotal and total rows are identified and excluded from the procurement list.
📚
Material Template Library Lookup (Gold flow)
Each extracted line item is matched against the Material Template Library — Alamco's verified product catalogue. The AI performs semantic matching: "gypsum ceiling tiles 600×600" maps to the catalogue SKU, brand, unit cost, and preferred supplier. For items with no direct match, the AI generates a structured procurement entry flagged for manual review. This lookup happens in parallel with extraction, not after.
⚖️
Confidence Scoring & Flagging
Every extracted item receives a confidence score (0–100%). Items above 90% proceed automatically. Items between 70–90% are processed but flagged amber for human review. Items below 70% are held and the procurement officer is notified to validate before the record is pushed to Odoo. This prevents incorrect purchase requisitions from entering the ERP system.
📐
Quantity Validation & Unit Harmonisation
The agent checks quantities for arithmetic consistency against BOQ subtotals where available. Units of measure are normalised to Alamco's standard set (e.g., "SQM" → "m²", "LM" → "m"). Quantities are also cross-checked against typical project scope ranges — an entry of 500,000 m² of floor tiles triggers an automatic outlier alert regardless of confidence score.
Output of this step: A fully structured, confidence-scored procurement list — every line item extracted, classified by division, matched to the catalogue where possible, with quantities validated and units normalised.

Node 3 · Reference · Gold flow
Material Template Library — Catalogue Intelligence
The Material Template Library is Alamco's living product and specification database — maintained and curated by the procurement team. The AI queries it in real-time during extraction to enrich every BOQ line item with verified procurement data.
🗃️
What the Library Contains
Every entry in the library includes: product name and description, internal SKU, category and subcategory, unit of measure, preferred brand(s), approved supplier list with contact details, current unit cost (updated from last PO), lead time (days), minimum order quantity, and applicable specifications or standards (e.g. BS/EN references). The library is version-controlled — every change is timestamped and attributed.
🔗
Semantic Matching Process
The AI uses vector embeddings to perform semantic similarity search across the library — not just keyword matching. "Ceramic floor tile 600×600 anti-slip", "anti-skid ceramic tile 60×60 cm", and "porcelain floor tile 600mm sq." all resolve to the same catalogue product despite different phrasing. This handles the real-world inconsistency of how materials are described across different BOQ sources and consultants.
✏️
Library Maintenance & Growth
When the procurement team manually maps or corrects a previously unmatched item, the library learns — the new mapping is saved. Over time, the match rate improves project by project. The team can also bulk-update the library by uploading a new product catalogue or price list; the AI merges it against existing entries, flagging conflicts for review rather than overwriting blindly.
Output of this step: Each BOQ line item enriched with: matched SKU, preferred supplier, unit cost, lead time, and minimum order quantity — directly feeding into the purchase requisition generated in the next step.
Vector semantic searchVersion-controlled catalogueAuto-enrichmentLearns from corrections

Node 4 · ERP Integration · Primary data flow
Odoo 19 ERP — Purchase Requisition Creation
Every validated procurement item is automatically pushed into Odoo 19's Purchase module as live records. No copy-paste. No re-entry. The ERP receives structured, verified data directly from the AI agent via API.
🔌
API Connection & Authentication
The system connects to Alamco's Odoo 19 instance via the official Odoo JSON-RPC API with OAuth2 authentication. The connection is persistent — no re-authentication per job. The AI agent operates under a dedicated service account with precisely scoped permissions: write access to Purchase Requisitions and read access to Products, Vendors, and UoM — nothing else.
🛒
Purchase Requisition Record Creation
For each approved procurement item, the agent creates a Purchase Requisition record in Odoo with: product (matched to Odoo's internal product catalogue), quantity, unit of measure, requested delivery date (calculated from project timeline and lead time), preferred vendor (from the approved list), estimated unit price (from library), project reference, and BOQ division tag. All fields are populated — no partial records.
🧩
Product Verification Against Odoo Catalogue
Before creating each record, the agent verifies the product exists in Odoo's product master. If the product is found, the Odoo internal ID is used directly. If not found (new product from the BOQ), the agent creates a draft product record and flags it for the procurement team to verify and activate — preventing orphan requisitions that reference non-existent products.
👤
Vendor Assignment
Each requisition is assigned to the preferred vendor from the Material Template Library, cross-checked against Odoo's vendor list. If the preferred vendor is not in Odoo, the system selects the next approved supplier or leaves the field as "To be assigned" — never blocks the record creation. Vendor selection logic can be configured per category (e.g., always use Vendor A for tiles above SAR 50,000).
📎
Original BOQ Attachment & Audit Trail
The original BOQ file is attached to the Odoo project record. Each Purchase Requisition includes a chatter note with: AI extraction timestamp, confidence score for this item, matched library entry, and the Codeox job ID. This creates a complete, queryable audit trail — from raw BOQ line to approved PO — entirely within Odoo.
Output of this step: Live Purchase Requisition records in Odoo 19 — product-matched, vendor-assigned, quantity-correct, and project-tagged. Ready for the procurement team to review and convert to RFQs or Purchase Orders.

Node 5 · Monitoring · Status feedback loop
Status & Delivery Tracker — Real-Time Monitoring
The AI agent's job doesn't end at requisition creation. It continuously monitors the lifecycle of every procurement record in Odoo — surfacing delays, escalating risks, and keeping the project team informed without manual chasing.
🔄
Continuous Odoo Sync
The tracker polls Odoo's Purchase module at configurable intervals (default: every 15 minutes) to read the current state of all active requisitions and purchase orders associated with projects managed through the system. Status changes — from Draft → Confirmed → Order Placed → In Transit → Received — are detected automatically and recorded in the tracker's timeline.
⏰
Delivery Date Intelligence
For each purchase order, the tracker calculates the expected delivery date from: order confirmation date + vendor lead time (from the library). If the scheduled delivery date passes without a received confirmation in Odoo, the item is automatically flagged as at-risk. The system escalates — first a notification to the procurement officer, then (if unresolved after 24h) an escalation to the project manager.
🔔
Automated Alerts & Escalations
Configurable alert rules fire based on procurement events: (1) Requisition sitting in Draft >48h — reminder to confirm; (2) PO not placed within 3 days of requisition approval; (3) Delivery expected within 5 days and not yet confirmed shipped; (4) Delivery overdue by any amount; (5) Item received with quantity variance vs PO. Alerts are sent via email and optionally via WhatsApp or SMS. Each alert links directly to the relevant Odoo record.
📊
Dashboard & Reporting
A live procurement dashboard (accessible inside the Codeox platform and embeddable in Odoo's portal) shows: all active projects and their procurement health (Green / Amber / Red), per-project item status breakdown, vendor performance (on-time delivery rate, avg. lead time by vendor), spend tracking vs BOQ estimate, and outstanding requisitions by age. Weekly PDF reports can be auto-generated and emailed to project managers.
🔁
Feedback Loop to AI Agent
Delivery performance data feeds back into the Material Template Library: if a vendor consistently delivers late, their lead time in the library is automatically adjusted upward for future BOQ processing. If a product's actual price on invoices consistently differs from the library estimate, the library unit cost is flagged for update. The system improves its own accuracy with each completed project cycle.
Output of this step: Continuous procurement visibility across every active project — delays surfaced before they impact site delivery, vendor performance data captured, and a self-improving feedback loop that makes every future BOQ processing cycle more accurate.
