Platform Overview · Architecture & Concurrent Incident Handling
Platform Overview
A multi-region platform built for scale and precision.
The Code-Ox Vision AI platform is engineered to operate at city-scale — 5000 cameras across multiple regions — while simultaneously maintaining sub-100ms alert latency and handling any number of concurrent incidents from the same camera frame. Both capabilities are architectural fundamentals, not add-ons.
Architecture · Multi-Region Scale

4-Tier Federated Architecture with Disaster Recovery

The platform uses a 4-tier multi-region federated architecture — the same codebase proven at 1000–2000 cameras, extended with a regional aggregation layer. ~240 edge nodes are distributed across 5 regions, feeding 25 regional aggregators, which connect to a central GPU cluster. A dedicated DR site provides full failover. Every tier scales horizontally without any architectural changes.

Concurrency · Parallel Incident Processing

All Incidents in a Frame, Captured Simultaneously

The detection pipeline uses single-pass inference — all objects in a frame are returned in one forward pass through the model. Each object is then evaluated against all configured rules in parallel. Every rule trigger produces its own atomic event with a unique ID, microsecond timestamp, and full metadata. Multiple incidents in the same frame produce multiple independent events — always, by design.

Design Principle

The 5000-camera tier extends the proven 3-tier architecture with a regional coordination layer. Horizontal scaling of node counts and cross-region synchronisation are the only additions — the core detection, rule evaluation, and event dispatch pipeline remains unchanged. Detailed subsystem documentation and BOM available on request.

5000-Camera Architecture · Four Tiers, Five Regions, One Platform
The 5000-camera tier extends the proven 3-tier architecture used at 1000–2000 cameras with a regional layer for multi-region coordination and a dedicated DR site. The architectural shape stays consistent — what changes is horizontal scaling of node counts and the addition of cross-region synchronisation.
Tier 00 · Source
Camera Fleet

5000 IP cameras across 5 regions. RTSP / ONVIF. Any brand, any age. Existing client infrastructure tapped via sub-stream — no rip-and-replace required. Cameras grouped by region for local processing proximity.

Tier 01 · Edge
Edge Fleet · ~240 Nodes

~240 edge nodes mixed fleet: 140 × Jetson AGX Orin Industrial (18–22 cam each) + 100 × x86 mini-server with RTX 4000 Ada (25–35 cam each). Detection, tracking, and rule evaluation run entirely at the edge — sub-100ms latency, no WAN dependency for alerting.

Tier 02 · Regional
Regional Aggregators · 25 Servers

25 aggregator servers (5 per region), each running RTX A5000 / RTX 4000 Ada with 256 GB DDR5 ECC RAM. Responsibilities: cross-camera person re-identification, regional event aggregation, regional event buffer for WAN-outage tolerance, and regional dashboard endpoints.

Tier 03 · Central + DR
Central GPU Cluster + DR

Primary: 16–20 × RTX PRO 6000 Blackwell 96GB + 6–8 × H200 141GB. 4–6 TB RAM, 3–5 PB tiered storage. VLM inference, semantic search, training, MLOps. DR site with 8 × RTX PRO 6000 + 2 × H200. RTO < 1 hr · RPO < 5 min. Async replication + streaming events.

Section 04
Concurrent Incident Handling · Multiple Incidents, Same Frame, Same Camera
This is one of the most common technical questions about Vision AI platforms — and it's worth answering properly, not casually. The platform is built around the assumption that multi-object multi-rule scenarios are normal, not edge cases.
The Short Answer
Yes. Always. Multiple incidents in the same frame produce multiple independent events — concurrently, atomically, with no missed detections and no race conditions.

Modern object detection models — RF-DETR, D-FINE, YOLO11/26 — are single-pass detectors. They don't detect "the first object," then "the second object." They take one frame as input and return all detected objects in a single inference call, each with its own bounding box, class label, and confidence score.

From that point forward, every detected object is treated independently. Each is evaluated against every configured rule in parallel. Each rule trigger becomes its own event. The architecture has no concept of "one event at a time per camera" — it's parallel by default.

Step-by-Step · What Happens When Multiple Incidents Occur in One Frame
1
Frame arrives at edge node

A single frame from the camera reaches the edge node at the configured rate (typically 5–15 FPS for AI inference, separate from the recording rate). The frame is timestamped with microsecond precision and passed to the detection pipeline.

2
Detection model runs once, returns all objects

The detection model performs a single forward pass on the frame. Output is a list of N detected objects — each with bounding box, class (person, vehicle, helmet, weapon, etc.), and confidence score. This happens in 8–25 ms on Jetson AGX Orin depending on model and frame complexity.

3
Tracker assigns persistent IDs across frames

BoT-SORT-ReID + CLIP-ReID associates each new detection with a previously-tracked object if one matches by motion + appearance. If 5 distinct people are in the frame, you get 5 distinct track IDs. Each person is a "tracked entity" carrying state across frames — useful for de-duplication later.

4
Rule engine evaluates all objects against all rules in parallel

For each detected object, the rule engine checks every configured rule simultaneously. Rules don't run sequentially. A typical set: is this person in polygon zone A? Is this person without PPE? Is person count above threshold? Is this person carrying a flagged item? Each is an independent boolean check, evaluated in microseconds.

5
Each rule trigger becomes an atomic event

Every (object, rule) pair that fires becomes its own event with: unique event ID (UUID v4), source camera ID, frame timestamp (microsecond precision), rule triggered, bounding box, track ID, confidence, severity. If 2 people enter restricted zone at once → 2 events. If one person triggers PPE violation AND restricted zone → 2 events (different rules). If 3 people fail PPE in one frame → 3 events.

6
Events flow to Kafka bus · downstream consumers process in parallel

Events are written to an internal Kafka event bus. Order is preserved per-camera (FIFO per camera ID), but processing across cameras is fully parallel. Downstream consumers — dashboard, alert router, voice agent, chat dispatcher, ERP webhook — subscribe independently. One slow consumer never blocks the others.

7
Alert clustering for forensic clarity

If a tracked person triggers the same rule on consecutive frames at 15 FPS, you'd get 15 alerts/second — useless. The clustering layer groups events by (track_id, rule_id, time_window) typically 30–60 seconds. One logical incident card appears on the dashboard, with all raw events accessible for forensic review. Critical: clustering only applies to same object × same rule. Two different people in a restricted zone = two separate clustered incidents — never collapsed into one.

8
Dashboard displays concurrent events with separate cards

Each clustered incident appears as its own card in the live event feed, ordered by severity and timestamp. Operators see exactly what happened: which person, which rule, which camera, which moment, with a thumbnail of the triggering frame. Cards can be acknowledged independently.

Section 04.2
Why This Architecture Avoids the Common Failure Modes
Five architectural concerns are commonly raised about concurrent incident handling in Vision AI systems. Each has a deliberate design answer — documented here with the specific mechanism that addresses it.
No "first object wins" because detection is single-pass

A common concern: "if the model sees the helmet violation first, does it miss the weapon?" The answer is no — modern single-pass detectors return all detections from one inference pass. There is no "first" detection. All objects are returned simultaneously from the same forward pass through the model.

No queueing per camera because rule evaluation is parallelised

Rules are evaluated as independent boolean checks across all detected objects. Five rules across three objects = fifteen parallel evaluations, completed in microseconds. There is no serial bottleneck. The rule engine is stateless per evaluation — no shared mutable state, no lock contention.

No missed events because the event bus accepts all dispatches atomically

Each rule trigger writes an event to the Kafka bus as an atomic operation. Even if the dashboard or downstream consumer is briefly slow, events accumulate in the bus and are processed in order without loss. The bus provides at-least-once delivery with deduplication at the consumer layer via event UUID.

No dashboard overload because clustering happens at the right layer

The clustering layer separates "raw events" (every frame's detections, written to the event store for forensics) from "incident cards" (one logical incident per track_id × rule × time-window). The operator sees actionable cards, not a flood of frame-level detections. The raw event record is always preserved for audit.

Cross-camera incidents handled at the Regional layer — not double-counted

If the same person walks across cameras 14 → 15 → 16 in sequence, BoT-SORT-ReID + CLIP-ReID at the Regional Aggregator stitches the track. Cross-camera tracking is a Regional layer responsibility, sitting above per-camera detection — so even concurrent multi-camera incidents are correctly attributed and not double-counted.
